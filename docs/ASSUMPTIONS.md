# Assumptions and Limitations (ข้อสมมติฐานและข้อจำกัดของระบบ)

ในการออกแบบสถาปัตยกรรมของโปรเจกต์ `Blog8byte` เราได้กำหนด **"กรอบการทำงาน (Boundaries)"** และ **"ข้อจำกัดที่ตั้งใจสร้างขึ้น (Self-Imposed Constraints)"** เพื่อให้โปรเจกต์เดินหน้าได้อย่างมีทิศทาง ไม่สะเปะสะปะ และตอบโจทย์ระดับ Production-Ready ได้ดีที่สุด

---

## 1. 🎨 Design Constraints (ข้อจำกัดด้านการออกแบบ)
- **The "No Clutter" Rule:** 
  - **ข้อจำกัด:** เราจะไม่อนุญาตให้ใช้ Component ที่มีเงาหนาๆ (Heavy Box-Shadows), ขอบโค้งมนจัดๆ (Rounded-Full), หรือสีสันที่ฉูดฉาดเกินไป (ยกเว้น Accent Colors ที่ตกลงกันไว้)
  - **เหตุผล:** เพื่อบังคับให้ UI ทั้งหมดอยู่ในสไตล์ "Earth Tone + Neo-Brutalist (Pirate Wires)" การมีข้อจำกัดนี้ทำให้เอกลักษณ์ของแบรนด์แข็งแกร่ง และโค้ด CSS (Tailwind) มีขนาดเล็กมาก

## 2. 🛡️ Security Strictness (ข้อจำกัดด้านความปลอดภัย)
- **Zero-Trust API (ไม่ไว้ใจ Client 100%):**
  - **สมมติฐาน:** ถือว่าข้อมูลทุกอย่างที่ส่งมาจากหน้าจอเว็บ (Frontend) หรือจากการยิง API ตรงๆ ล้วนเป็นข้อมูลที่ "อันตรายและอาจผิดพลาดได้"
  - **ข้อจำกัด:** API Route ทุกเส้นในโปรเจกต์ **ห้าม** ประมวลผลข้อมูลทันที จะต้องผ่านด่าน `Zod Validation` เพื่อตรวจจับ Type และผ่าน `withErrorHandler` (HOF) ทุกครั้ง แม้ว่ามันจะทำให้ต้องเขียนโค้ดเพิ่มขึ้นนิดหน่อย (Boilerplate) แต่แลกมาด้วยระบบที่ไม่พังตอนรันจริง (Runtime Safety)

## 3. 📝 Documentation Policy (ข้อจำกัดเรื่องการเขียน Docs/Comments)
- **Selective JSDoc (เขียนเฉพาะจุดที่จำเป็น):**
  - **ข้อจำกัด:** เราจะ **ไม่เขียน JSDoc ให้กับทุกๆ ฟังก์ชันหรือ Component** แบบหว่านแห แต่จะบังคับเขียนเฉพาะในโฟลเดอร์ `src/lib/` หรือ Utilities ส่วนกลางเท่านั้น
  - **เหตุผล:** เพื่อรักษาความเร็วในการพัฒนา (Developer Velocity) โค้ดของ React Component ควรจะอ่านแล้วเข้าใจได้ทันทีจากชื่อตัวแปร (Self-Documenting) การบังคับเขียนคอมเมนต์ทุกจุดคือการ Over-engineering ที่ไร้ประโยชน์ในระยะยาว

## 4. 🗄️ Database & Infrastructure Assumptions (ข้อสมมติฐานด้านฐานข้อมูล)
- **Relational First:**
  - **สมมติฐาน:** โครงสร้างข้อมูลของ Blog (User, Posts, Comments) มีความสัมพันธ์กันอย่างชัดเจน
  - **ข้อจำกัด:** เลือกล็อกสเปคที่ **PostgreSQL** (ผ่าน Drizzle ORM) เท่านั้น จะไม่ซัพพอร์ต NoSQL (เช่น MongoDB) เพื่อดึงประสิทธิภาพของ Type-Safe SQL ออกมาให้สูงที่สุด และสมมติฐานว่าใน Production จะใช้ Serverless Postgres (เช่น Neon หรือ Vercel Postgres) เพื่อลดภาระการดูแลเซิร์ฟเวอร์

## 5. ⚡ Performance Assumption (ข้อสมมติฐานด้านความเร็ว)
- **Read-Heavy System:**
  - **สมมติฐาน:** บล็อกคือระบบที่มีคน "เข้ามาอ่าน" มากกว่า "เข้ามาเขียน" หลายร้อยเท่า
  - **ข้อจำกัดทางการออกแบบ:** สถาปัตยกรรมจะเน้นไปที่ React Server Components (RSC) เป็นหลัก เพื่อเรนเดอร์ HTML ให้เสร็จจากฝั่งเซิร์ฟเวอร์ (SSR/SSG) และส่ง Javascript ไปให้เครื่องผู้ใช้น้อยที่สุดเท่าที่จะทำได้ (ลด Client-side rendering) เพื่อให้หน้าเว็บโหลดเสร็จในเสี้ยววินาทีแม้เน็ตจะช้า

## 6. 🇹🇭 Localization & Data Integrity (ข้อจำกัดด้านภาษาและการตรวจสอบข้อมูล)
- **Thai Language & Numeric Validation:**
  - **สมมติฐาน:** เนื่องจากเป็นโปรเจกต์สำหรับประเทศไทย ข้อมูลฟอร์ม (Form Inputs) จำเป็นต้องรองรับภาษาไทยและตัวเลขได้อย่างถูกต้อง
  - **ข้อจำกัด:** จะมีการเขียน Custom Zod Schema พิเศษร่วมกับ Regex ในการตรวจสอบอักขระภาษาไทย (Thai characters) และชุดตัวเลข เพื่อป้องกันความผิดพลาดและข้อมูลขยะ

## 7. 👮 Content Moderation & Access Control (ข้อจำกัดด้านการคัดกรองเนื้อหา)
- **Admin Approval Workflow:**
  - **สมมติฐาน:** ระบบ Blog ตาม Assignment ระบุว่าอาจจะมีการคัดกรองหรือจัดการสถานะข้อมูล (เช่น รอการอนุมัติ)
  - **ข้อจำกัด:** สถาปัตยกรรม Database (Drizzle schema) จะต้องถูกออกแบบให้มีคอลัมน์สถานะ (เช่น `status: 'pending' | 'approved' | 'rejected'`) ตั้งแต่ต้น และวางโครงสร้างสำหรับ Role-Based Access Control (RBAC) เพื่อให้แอดมินเท่านั้นที่มีสิทธิ์กดอนุมัติ

## 8. 📦 Tooling & Local Infrastructure (ข้อจำกัดด้านเครื่องมือและการจำลองสภาพแวดล้อม)
- **Strict Package Manager (`pnpm`):**
  - **ข้อจำกัด:** ล็อกการใช้ `pnpm` เท่านั้น (เปิดการทำงานผ่าน `corepack`) เพื่อรับรองประสิทธิภาพการ Resolve dependency ที่รวดเร็วและใช้พื้นที่ดิสก์น้อยที่สุด (อิงจาก Q2-T3-Decision)
- **Containerized Local Database:**
  - **สมมติฐาน:** ทีมหรือกรรมการผู้ตรวจต้องสามารถติดตั้งฐานข้อมูลและรันแอปได้ในคำสั่งเดียว
  - **ข้อจำกัด:** บังคับใช้ `docker-compose.yml` ในการเปิด Local PostgreSQL เท่านั้น ไม่สนับสนุนการลง Database engine แบบดิบๆ ลงบนเครื่องส่วนตัว เพื่อลดปัญหา "It works on my machine"
